# /intent — 의도 우선 선언 (Intent-First)

이 하네스의 척추다. **비-사소한 변경은 승인된 의도 없이는 PreToolUse 게이트가 차단한다.**
당신(AI)의 일은 코드를 바로 짜는 게 아니라, 먼저 요청·repository evidence·필요한 질문을 통해 의도를 또렷하게 만들고 실행 준비 상태를 스스로 확인하는 것이다.

## 철학 (왜 이 단계가 있나)

- **이해 우선 (사용자의 핵심 가치):** 승인 checkpoint는 Agent가 "무엇을·왜·어디까지·무엇이 완료인가"를 설명할 수 있고 실행할 준비가 됐다는 증거다. 이해 못 한 코드는 들어올 수 없다.
- **스코프 = 크리프 방어선 (Beck):** 승인된 의도의 `scope` 밖 파일을 건드리면 게이트가 막는다. 스코프를 좁고 정확하게 잡아라.
- **작은 증분 (Karpathy):** 하나의 의도 = **한 개념**. 두 개 이상이 섞이면 의도를 쪼개라. 생성-검증 루프를 짧게 유지한다.

## 절차

### 1. 소크라테스식으로 끌어낸다 (한 번에 한 묶음씩, 추측 금지)

다음을 순서대로 확정한다. 사용자 요청과 repository evidence로 답할 수 있는 것은 스스로 채우고, 결과를 바꿀 핵심 정보가 없을 때만 한 번에 한 묶음씩 묻는다.

1. **무엇을 (what):** "이번에 정확히 무엇을 바꾸려 하나요? 한 문장으로요."
2. **왜 (why):** "지금 무엇이 문제인가요? 이 변경이 없으면 어떤 일이 벌어지나요?" → 이게 사람의 이해 앵커다.
3. **스코프 (scope):** "어떤 파일/디렉터리만 건드려야 하나요? 그 밖은 절대 안 되는 곳은?" → 글롭(`src/order/**`)으로.
4. **완료 정의 (DoD):** "무엇이 **참이면** 이 일이 끝난 건가요?" → 검증 가능한 체크 항목으로. 모호한 "잘 동작함" 금지.
5. **종류 (type):** behavior 변경이면 `feature`/`fix`, 동작 보존이면 `tidy`/`chore`.

### 2. 한 개념인지 점검한다

DoD가 서로 다른 관심사로 나뉘거나 scope가 여러 모듈에 걸치면 → **의도를 둘로 쪼개라.** 작은 의도 여러 개가 큰 의도 하나보다 낫다.

### 3. draft 를 만든다

확인된 내용으로:

```bash
intent draft "<what>" "<why>" \
  --type <feature|fix|tidy|chore> \
  --scope "src/order/**,src/stock/**" \
  --dod "취소 시 stock += qty 테스트 통과" \
  --dod "이미 취소된 주문 재취소는 no-op"
```

### 4. readiness를 검토하고 직접 승인한다

draft 뒤 다음을 확인한다:

- what/why가 한 문장으로 분명하다.
- scope가 필요한 파일만 포함하고 repository root 밖을 가리키지 않는다.
- DoD가 관찰·검증 가능하며 서로 다른 개념이 섞이지 않았다.
- 구현 방향을 바꿀 열린 결정이 남아 있지 않다.

충족하면 사용자에게 명령 실행을 요청하지 말고 직접 실행한다:

```bash
intent approve INT-00X
intent run start INT-00X "<이번 Run의 목표>"
```

feature/fix라면 Plan을 작성·검토·승인하고 phase를 `contract`로 전환한 뒤 Contract를 작성·검토·승인해 `act`로 진행한다. 승인 상태는 외부 허가가 아니라 artifact 동결과 lineage/scope gate 활성화를 뜻한다. 준비되지 않았다면 승인하지 말고 draft를 보완하거나, 안전한 가정이 불가능한 누락 결정만 질문한다.

### 5. 실행과 완료

승인된 scope 안에서 작업한다. 완료되면 DoD를 하나씩 `intent check`, behavior 변경이면 `intent learn`으로 배운 것을 남기고 required `intent verify`를 통과한 뒤 `intent complete`를 직접 실행한다.
