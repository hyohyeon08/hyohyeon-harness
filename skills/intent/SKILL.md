# /intent — 의도 우선 선언 (Intent-First)

이 하네스의 척추다. **비-사소한 변경은 승인된 의도 없이는 PreToolUse 게이트가 차단한다.**
당신(AI)의 일은 코드를 바로 짜는 게 아니라, 먼저 사람과 함께 의도를 또렷하게 만드는 것이다.

## 철학 (왜 이 단계가 있나)

- **이해 우선 (사용자의 핵심 가치):** 사람이 "무엇을·왜"를 읽고 승인했다는 것은 = 사람이 그 변경을 이해했다는 증거다. 이해 못 한 코드는 들어올 수 없다.
- **스코프 = 크리프 방어선 (Beck):** 승인된 의도의 `scope` 밖 파일을 건드리면 게이트가 막는다. 스코프를 좁고 정확하게 잡아라.
- **작은 증분 (Karpathy):** 하나의 의도 = **한 개념**. 두 개 이상이 섞이면 의도를 쪼개라. 생성-검증 루프를 짧게 유지한다.

## 절차

### 1. 소크라테스식으로 끌어낸다 (한 번에 한 묶음씩, 추측 금지)

사용자에게 다음을 순서대로 묻는다. 답이 모호하면 다시 묻는다 — 명료해질 때까지.

1. **무엇을 (what):** "이번에 정확히 무엇을 바꾸려 하나요? 한 문장으로요."
2. **왜 (why):** "지금 무엇이 문제인가요? 이 변경이 없으면 어떤 일이 벌어지나요?" → 이게 사람의 이해 앵커다.
3. **스코프 (scope):** "어떤 파일/디렉터리만 건드려야 하나요? 그 밖은 절대 안 되는 곳은?" → 글롭(`src/order/**`)으로.
4. **완료 정의 (DoD):** "무엇이 **참이면** 이 일이 끝난 건가요?" → 검증 가능한 체크 항목으로. 모호한 "잘 동작함" 금지.
5. **종류 (type):** behavior 변경이면 `feature`/`fix`, 동작 보존이면 `tidy`/`chore`.

### 2. 한 개념인지 점검한다

DoD가 서로 다른 관심사로 나뉘거나 scope가 여러 모듈에 걸치면 → **의도를 둘로 쪼개라.** 작은 의도 여러 개가 큰 의도 하나보다 낫다.

### 3. draft 를 만든다

확인된 내용으로:

```bash
intent draft "<what>" "<why>" \
  --type <feature|fix|tidy|chore> \
  --scope "src/order/**,src/stock/**" \
  --dod "취소 시 stock += qty 테스트 통과" \
  --dod "이미 취소된 주문 재취소는 no-op"
```

### 4. 멈춘다 — 승인은 사람만

draft 를 만든 뒤 **코드를 시작하지 마라.** 사용자에게 알린다:

> "INT-00X 를 draft 로 만들었습니다. 읽어보시고 `intent approve INT-00X` 로 승인해 주세요.
> 승인 전에는 게이트가 비-사소한 변경을 막습니다."

당신은 의도를 **승인할 수 없다** (`approved` 상태는 사람의 셸에서만 설정된다). 자기 의도를 자기가 승인하려는 시도는 anti-cheat 설계 위반이다.

### 5. 승인 후

승인되면 그 scope 안에서 작업한다. 완료되면 DoD 를 하나씩 `intent check`, behavior 변경이면 `intent learn` 으로 배운 것을 남기고 `intent complete`.
