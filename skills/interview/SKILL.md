# /interview — 백그라운드 GAP 줄이기

목표: 코드를 짜기 전에 **사람의 머릿속 모델 = AI의 모델** 로 맞춘다.
단순 요구사항 정리가 아니라, **사람이 당연하게 여겨 말하지 않은 것(암묵지)** 을 끌어내는 게 핵심이다.

## 원칙: AI가 모르는 걸 모른다 (unknown-unknowns)

추상적 목표를 받으면 바로 구현하지 마라. GAP은 대개 *말해지지 않은 전제* 에 숨어 있다.

## 두 축으로 캐묻는다

### 축 1 — 가정을 명시하고 교정받기
당신(AI)이 세운 가정을 **눈에 보이게 나열**하고, 사람에게 틀린 걸 짚어달라 한다.

> "제가 이렇게 가정하고 있습니다. 틀린 게 있으면 알려주세요:
>  1. 인증은 기존 세션 토큰을 재사용한다
>  2. 이 API는 단일 테넌트다
>  3. 실패 시 조용히 무시가 아니라 예외를 던진다 …"

사람이 하나라도 "그건 아니다" 하면 → 거기에 GAP이 있다. 더 파고든다.

### 축 2 — '당연한 것'을 역질문
사람이 생략했을 법한 암묵지를 직접 물어라:

- "이 도메인/팀에서 **문서엔 없지만 당연한** 규칙이 있나요?"
- "이건 **절대 하면 안 되는** 게 있나요? (건드리면 안 되는 코드, 깨면 안 되는 계약)"
- "과거에 **이런 걸로 사고 난 적** 있나요?" (→ 나중에 `/postmortem` 후보)
- "성공/실패를 **어떻게 눈으로 확인**하나요?"
- 경계 예시로 확인: "**이건 되고 저건 안 된다**, 맞나요?"

한 번에 한 묶음씩. 모호하면 명료해질 때까지 다시 묻는다 (추측 금지).

## 산출물: 공유 이해 문서(spec) → 위키 + 사람 승인

대화가 휘발되면 가치가 없다. 합의된 이해를 spec 으로 남긴다:

```bash
intent spec draft "주문 취소 흐름"
intent wiki append spec-주문-취소-흐름 "## 목표\n..."
intent wiki append spec-주문-취소-흐름 "## 가정(확정)\n- ...\n## 명시적 비목표\n- ...\n## 경계 예시\n- ..."
```

그리고 **멈춘다**:

> "spec 을 위키에 정리했습니다. 읽어보시고 제 이해가 맞으면 `intent spec approve spec-주문-취소-흐름` 로 승인해 주세요."

승인은 **사람만** 한다 (Claude Code/Codex AI 셸에선 거부됨). 승인된 spec 은 위키에 남아 다음 세션과 intent skill이 참조한다 — 지식이 복리로 쌓인다.

## 다음 단계
spec 이 승인되면 `/intent` 로 넘어가 실제 의도를 선언한다. spec = 공유 이해, intent = 이번에 짤 작은 증분.
