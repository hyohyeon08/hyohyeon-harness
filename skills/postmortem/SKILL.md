# /postmortem — 실패 → 재발 방지

같은 실패를 두 번 겪지 않도록 원장에 남긴다. 핵심은 **분기**다:

- **표현 가능한 실패** (특정 파일 금지, 특정 패턴 금지) → **게이트 규칙 초안** + 위키 기록
- **그 외** (판단·맥락이 필요한 실패) → **위키 기록만**

## 3가지 트리거

1. **사람이 교정했을 때** — 대화에서 "그게 아니라…" 류의 교정이 나오면, 무엇을 잘못 가정했는지 포스트모템으로 남긴다.
2. **테스트 실패 / 버그** — 빨간 테스트나 런타임 에러의 근본 원인을 찾았으면 기록한다.
3. **수동 호출** — 사람이 명시적으로 요청.

## 절차

1. **무엇이·왜**를 한 문장으로: 무엇이 잘못됐고, **근본 원인**이 뭔가.
2. **재발 방지**를 적는다: 다음엔 무엇을 다르게 해야 하나.
3. **게이트로 표현 가능한가?** 판단한다:
   - "src/legacy 는 절대 수정 금지" → `forbid-path`
   - "console.log 금지", "@Transactional 제거 금지" → `forbid-pattern`
   - 표현 가능하면 `--rule` 로 규칙 초안을 함께 만든다 (사람이 나중에 `intent rule approve`).
4. 기록한다:
   ```bash
   intent postmortem "취소 시 재고 복원 누락" \
     --cause "OrderService.cancel 이 StockService 를 호출 안 함" \
     --prevent "취소 경로에 재고 복원 통합 테스트를 항상 둔다" \
     --rule forbid-pattern "cancel\\(\\)\\s*\\{[^}]*\\}" 
   ```
   (규칙이 애매하면 `--rule` 없이 위키 기록만)

## 원칙

- 규칙(강제)과 위키(읽는 지식)는 분리된 채널이다. 규칙은 **사람 승인** 전까지 효력 없다 (anti-cheat).
- 포스트모템은 비난이 아니라 **시스템 개선**이다 — 실패할 때마다 환경에 영구 수정을 새긴다.
